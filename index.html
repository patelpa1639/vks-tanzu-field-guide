"WTF is VKS?" — The 5-Minute Explainer

▶

Core Concept

VKS is VMware's enterprise Kubernetes runtime built directly into vSphere/VCF. It lets customers run containers at the same level as VMs — same policies, same security, same management plane. No separate K8s distro to install or manage.

The Naming Journey

⚠ Critical context — customers are confused by naming changes
  • 2020–2023: "vSphere with Tanzu" / "Tanzu Kubernetes Grid Service" (TKGs)
  • Oct 2024 (VKS 3.2): TanzuKubernetesCluster (TKC) API deprecated
  • Mar 2025 (VKS 3.3): Official rebrand to "vSphere Kubernetes Service (VKS)" and "vSphere Kubernetes release (VKr)" replacing TKr
  • Now: VKS is the name. Period. If customer says "Tanzu" they probably mean VKS unless they specifically mean Tanzu Platform (the PaaS).

VKS vs Tanzu Platform

  • VKS = Kubernetes runtime/orchestration service INCLUDED in VCF. It's the infrastructure layer. Think "managed K8s on vSphere."
  • Tanzu Platform = The PaaS layer that sits ABOVE VKS. Application lifecycle, service mesh, build services, developer experience. Can run on any K8s, not just VKS.
  • Customer analogy: VKS is like EKS (the runtime), Tanzu Platform is like AWS App Runner (the developer platform on top).

Architecture Stack

Tanzu Platform (PaaS) — Optional, separate product VKS Workload Clusters — Your apps run here (Cluster API v1beta1) vSphere Supervisor — K8s control plane in vSphere (3 CP VMs/cluster) ┌─────────────┬─────────────┬─────────────┐ │ vSphere │ Virtual │ VKS │ │ Namespaces │ Machine │ Service │ │ (RBAC + │ Service │ (Cluster │ │ quotas) │ │ lifecycle) │ └─────────────┴─────────────┴─────────────┘ VMware Cloud Foundation (VCF) — vSphere + vSAN + NSX + SDDC Manager Hardware: Lenovo ThinkAgile VX Series + XClarity Integrator for vCenter

How VKS Actually Works

  1. Admin enables vSphere Supervisor on a vSphere cluster (deploys 3 control plane VMs)
  2. Admin creates vSphere Namespaces — resource boundaries with RBAC, storage policies, compute quotas, and VM classes
  3. Developer requests a VKS Cluster by applying a YAML manifest (Cluster API v1beta1) to the Supervisor
  4. VKS provisions the cluster: control plane + worker nodes as VMs, wired to vSAN storage, NSX/Avi networking, specified VKr version
  5. Developer gets a kubeconfig and uses standard kubectl — it's a conformant, upstream K8s cluster
  6. VKS handles lifecycle: upgrades, scaling (including scale-to-zero), node rotation, certificate management

vSphere Pods (VCF with NSX only)

Special mode: K8s pods run directly on ESXi using a "Spherelet" — essentially a micro-VM per pod. No guest cluster needed. Very efficient, but requires NSX networking. Not available in VVF (vSphere Foundation) deployments.

VKS 3.6 — What's New

▶

Announced at KubeCon EU Amsterdam, March 23, 2026. This is the latest release — know every feature and how to position it.

Kubernetes 1.35 Support

  • CNCF-certified, conformant K8s
  • 24-month extended support per version with overlapping version support
  • Customers can upgrade on their own schedule
  • Notable 1.35 features: configurable maxUnavailable for StatefulSet rolling updates, improved topology awareness for latency-sensitive workloads

RHEL 9 Build Your Own Image (BYOI)

  • Huge for enterprise customers locked into Red Hat ecosystems
  • New ImageBaker tool (replaces Image Builder) integrated with VCF CLI
  • Supports RHEL 9, Photon OS 5, Ubuntu 22.04/24.04, Windows Server 2022
  • Heterogeneous clusters: different OS per node pool
  • ImageBaker is standalone (no vCenter dependency), uses OCI-based BuildKit
  • Customers keep their Red Hat licensing and support chain intact

TuneDProfile — Declarative Kernel Tuning

  • CRD for OS-level tuning via standard K8s YAML
  • Per-node-pool granularity — tune database pools differently from app pools
  • Built-in builtin-vks-v3.6.0 profile solves 90% of common issues:
    • vm.max_map_count = 262144 (Elasticsearch/SonarQube)
    • kernel.pid_max >= 4194304 (PID exhaustion on dense nodes)
    • Network ring buffer optimization
  • Custom profiles for Redis, Kafka, MongoDB, Telco workloads
  • Automated drain-and-reboot handling for kernel parameter changes
  • Replaces fragile init containers and DaemonSet hacks

AppArmor Profile Management

  • Define AppArmor profiles as Custom Resources
  • Auto-sync across all worker nodes or specific node pools
  • Granular security without locking clusters into rigid hardening

Open, Extensible Networking

  • Supported path for third-party CNI plugins (not just Antrea)
  • New partnerships: F5 BIG-IP CIS, Kong Gateway, Tigera Calico
  • CNI selection during cluster creation via bootstrapAddons variable (VKr 1.35+)
  • ClusterBootstrap for CNI is deprecated for 1.35+

NGINX Deprecation → Avi Gateway API

⚠ TAM Action Required: NGINX Ingress Controller deprecated upstream in K8s 1.35. Proactively discuss Avi Gateway API migration path with customers using NGINX.

Upgrade Safety Improvements

  • Pre-upgrade checks now detect misconfigured admission webhooks (not just PDBs)
  • Checks run continuously, exposed via SystemCheckSucceeded Condition
  • Identifies blockers BEFORE maintenance windows, not during

nftables Backend for kube-proxy

  • Better performance and scalability for network policy enforcement
  • Replaces iptables-based kube-proxy

VKS Support Bundle Tool

  • Cluster users can generate support bundles independently
  • No longer requires vCenter SSO Administrator privileges

Centralized Firewall Management

  • API-driven management of node-level firewall rules across all supported OSes
  • Open required ports for HostPorts and NodePort Services through cluster configuration
  • Replaces privileged init containers or DaemonSets for port management

The Three-Layer Upgrade Model

⚠ This confuses everyone — make sure you can explain it clearly
  1. Supervisor Upgrade — Least frequent. In vSphere 8.x: delivered via vCenter patches. In vSphere 9.x: decoupled from vCenter, uses content library. Must be on Supervisor K8s 1.30+ for VKS 3.6.
  2. VKS Version Upgrade — More common (~every 4 months, offset from upstream by ~2 months). Downloaded from Broadcom support portal as a package YAML. Registered and activated on the Supervisor. Delivers new K8s versions AND new ClusterClass templates.
  3. VKr (vSphere Kubernetes Release) Upgrade — Per-cluster. Upgrading the actual Kubernetes version on your workload clusters. Independent of the other two. Rolling update of nodes.
⚠ Critical upgrade rule: VKS 3.6 drops VKr 1.31 compatibility. Before upgrading to VKS 3.6, ALL clusters must be on VKr 1.32 or later.

TKC → Cluster API Migration

⚠ #1 Action Item for existing VKS/TKGs customers
  • TanzuKubernetesCluster (TKC) API is deprecated since VKS 3.2
  • TKC cannot create clusters with K8s v1.33 or later
  • VKS 3.3 introduced streamlined migration tooling
  • To upgrade existing clusters to v1.33+, you MUST first migrate off TKC API
  • Known issue: retiring a TKC with Additional Trusted CA certificates requires a workaround (fixed in VKS 3.3.3)
  • TAM action: ask EVERY customer if they're still on TKC API and start migration planning NOW

Hands-On: Key VKS Workflows

▶

Login to Supervisor (legacy kubectl-vsphere)

bash
kubectl-vsphere login \
  --insecure-skip-tls-verify \
  --vsphere-username administrator@vsphere.local \
  --server=<supervisor-api-server-ip> \
  --tanzu-kubernetes-cluster-namespace=<namespace> \
  --tanzu-kubernetes-cluster-name=<cluster-name>

Login via VCF CLI (VCF 9 / modern method)

bash
# Download kubeconfig from vCenter UI or VCF CLI
# Then use --kubeconfig flag
kubectl --kubeconfig=<path-to-kubeconfig> get nodes

Create a VKS Cluster (Cluster API v1beta1)

yaml
apiVersion: cluster.x-k8s.io/v1beta1
kind: Cluster
metadata:
  name: my-workload-cluster
  namespace: my-namespace
spec:
  clusterNetwork:
    pods:
      cidrBlocks: ["192.168.0.0/16"]
    services:
      cidrBlocks: ["10.96.0.0/12"]
  topology:
    class: tanzukubernetescluster  # or latest builtin-generic class
    version: v1.35.0+vmware.1
    controlPlane:
      replicas: 3
      metadata:
        annotations:
          run.tanzu.vmware.com/resolve-os-image: os-name=ubuntu
    workers:
      machineDeployments:
        - class: node-pool
          name: worker-pool-1
          replicas: 3
          metadata:
            annotations:
              run.tanzu.vmware.com/resolve-os-image: os-name=ubuntu
    variables:
      - name: vmClass
        value: best-effort-large
      - name: storageClass
        value: vcf-default-storage-policy

Apply and Check Cluster

bash
kubectl apply -f cluster.yaml
kubectl get cluster -n my-namespace
kubectl get machines -n my-namespace

Check VKS Cluster Readiness Conditions

bash
kubectl get cluster my-workload-cluster -n my-namespace -o yaml | grep -A 20 "conditions:"

Scale Worker Nodes

bash
# Edit the cluster spec
kubectl edit cluster my-workload-cluster -n my-namespace
# Change workers.machineDeployments[0].replicas to desired count

TuneDProfile Example (VKS 3.6)

yaml
apiVersion: run.tanzu.vmware.com/v1alpha1
kind: TuneDProfile
metadata:
  name: high-perf-database
  namespace: my-namespace
spec:
  profile: |
    [main]
    summary=High performance database profile
    include=builtin-vks

    [sysctl]
    vm.max_map_count=262144
    vm.swappiness=1
    vm.overcommit_memory=1
    net.core.somaxconn=65535
    net.ipv4.tcp_max_syn_backlog=65535

List Available VKr Versions

bash
kubectl get kubernetesreleases
# or for legacy:
kubectl get tanzukubernetesreleases

Common Customer Questions

▶
▶ "We're still on TKC API. What do we need to do?"
You need to migrate to Cluster API (v1beta1) before upgrading to K8s v1.33 or later. VKS 3.3 introduced a streamlined migration mechanism. The TKC API is deprecated and will be removed. Start migration planning now — this is the #1 action item for any existing VKS/TKGs customer.
▶ "How is VKS different from Tanzu?"
VKS is the Kubernetes runtime built into VCF — it provisions and manages K8s clusters on vSphere infrastructure. Tanzu Platform is a separate PaaS product that provides developer experience, application lifecycle, and service mesh capabilities on top of any Kubernetes. You can use VKS without Tanzu Platform, and many customers do.
▶ "What about Broadcom licensing changes?"
VKS is included in VMware Cloud Foundation (VCF) licensing. There is no separate VKS license. VCF is sold as a 3-year minimum subscription, licensed per core (minimum 16 cores per CPU, 72 per order). If the customer is on VCF, they already have VKS. Tanzu Platform features may require additional licensing.
▶ "Can we run VKS air-gapped?"
Yes. VKS supports air-gapped deployments natively. You manually download VKS packages and VKr images from the Broadcom support portal, host them in a local content library, and register them with vCenter. The new ImageBaker tool in VKS 3.6 also operates standalone without vCenter dependency.
▶ "What about GPU and AI/ML workloads?"
VKS supports GPU passthrough for AI/ML workloads. VCF 9 with ThinkAgile VX supports NVIDIA GPU-dense configurations (ThinkSystem SR675 V3 with up to 8 GPUs). VKS earned CNCF AI Conformance certification. Standard packages support frameworks like TensorFlow, PyTorch, and Kubeflow on VKS clusters.
▶ "NGINX Ingress is deprecated — what do we use?"
Kubernetes 1.35 officially deprecates the NGINX Ingress Controller. Broadcom recommends migrating to the Avi Gateway API, which provides native integration with VKS and the Avi load balancer. F5 BIG-IP CIS is also now a validated option for customers with existing F5 investments.
▶ "How do we handle database workloads on K8s?"
VKS 3.6 introduces TuneDProfiles — declarative kernel tuning as a CRD. The built-in profile fixes common issues (vm.max_map_count for Elasticsearch, PID limits for dense workloads). You can create custom profiles per node pool for Redis, Kafka, MongoDB, etc. This replaces hacky init containers and DaemonSets.
▶ "Why VKS over just installing upstream Kubernetes ourselves?"
Lifecycle management. VKS handles provisioning, upgrades, scaling, certificate rotation, networking integration (NSX/Avi), storage integration (vSAN CNS), RBAC (vSphere Namespaces), and monitoring — all through the VCF platform your team already manages. Upstream K8s gives you day-0. VKS gives you day-0 through day-2+.
▶ "What OS options do we have for nodes?"
As of VKS 3.6: Photon OS 5, Ubuntu 22.04 and 24.04, Windows Server 2022, and RHEL 9 (via BYOI / ImageBaker). You can run heterogeneous clusters with different OS per node pool. Ubuntu Pro subscription is required for Ubuntu FIPS mode.
▶ "How does VKS integrate with our existing Lenovo XClarity management?"
XClarity Integrator for VMware vCenter and VCF Operations provides unified hardware-to-workload visibility. Lenovo hardware health, firmware updates, and configuration management appear directly in vCenter alongside your VKS clusters and VMs. It's a single pane of glass from bare metal to containers.
